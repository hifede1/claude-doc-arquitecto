Tablero de bloques

Las sesiones del plan de construcción (FICHA §7) son los bloques de esta obra. Tick ámbar = marcado sin verificar; verde solo tras re-auditoría en código.

Cómo se usa: abrí la ficha de un bloque → «Copiar encargo» → pegalo en una sesión de Claude Code (o dejá que /orquestar tome el issue). Al terminar, marcá el tick (queda ámbar) y pedí «actualizá el tracker»: la re-auditoría verifica EN CÓDIGO y recién ahí pasa a verde. En modo orquestado el equivalente del ámbar es el PR esperando tu firma.

Stack por capas

Cliente

  • Claude Code CLI — ejecuta los comandos; los plugins cargan al inicio de sesión, namespaceados (/doc-arquitecto:documentar).
  • Sistema de plugins — marketplace fede-tools (source: directorio local). Caché instalada: v0.2.0 (actualizada 2026-07-17). ⚠️ Gotcha permanente: la caché NO se sincroniza sola — actualizar tras cada cambio de comandos.

Datos

  • Git / GitHub — hifede1/claude-doc-arquitecto ⚠️ privado hasta S05: la instalación remota de terceros falla. Issues #2–#5 = cola de trabajo; PRs = canal de firma.
  • localStorage — ticks ámbar de este tracker (viven en UN navegador; el tracker es mapa, no despachador).

Servicios

  • gh CLI — dos cuentas en esta máquina: hifede1 (este repo) y estebaproject. ⚠️ Verificar la cuenta activa antes de operar.

Calidad

  • CI validate.yml — 3 checks estructurales (JSON, comandos, consistencia). Sin checks de contenido de comandos (deuda menor aceptada).
  • Protocolo de corridas — los criterios de la FICHA se verifican con corridas reales en repos de prueba (§6), no con tests unitarios: el producto son prompts.
